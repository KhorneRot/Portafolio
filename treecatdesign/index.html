Contexto breve

Una app para registrar, entender y anticipar el dinero del mes.

Hormigon nace de una necesidad real: controlar finanzas personales desde Excel funcionaba, pero actualizar gastos, ingresos extra, deudas familiares y compras a meses se volvió tedioso.

La app está pensada para personas que ya intentan organizarse, pero necesitan registrar movimientos desde el celular, revisar cómo va el mes y anticipar pagos sin depender de una computadora.

Problema + oportunidad

Las herramientas manuales ayudan a organizarse, pero fallan cuando registrar dinero exige demasiado esfuerzo justo en momentos cotidianos.

¿Cómo podríamos hacer que revisar y registrar finanzas sea tan inmediato como consultar el celular?

01

Investigación ligera

Qué observé antes de diseñar.

Validé el problema con uso real en casa, conversaciones informales y síntesis de patrones. El objetivo era separar una molestia personal de una fricción repetible.

Contexto Uso real en casa

Observé cuándo se rompía el hábito: al dejar compras pendientes para capturarlas después, al mezclar deudas con gastos del mes y al depender del escritorio.

Validación Conversaciones informales

Contrasté el problema con amigos y familiares que usan Excel, notas o apps bancarias aisladas para entender qué soluciones ya intentaban.

Síntesis Patrones de comportamiento

Agrupé las respuestas por fricciones: captura tardía, falta de visibilidad del mes, ingresos variables y dificultad para anticipar pagos futuros.

Insights clave

El problema no era falta de intención: era esfuerzo, momento y claridad.

Insight 01

La fricción mata el hábito.

Cuando el registro depende de abrir una hoja de cálculo, el usuario pospone la captura y termina perdiendo el control del mes.

Insight 02

El dinero se mueve en momentos pequeños.

Los gastos, préstamos familiares e ingresos extra ocurren durante el día; la herramienta debía estar disponible justo en ese momento.

Insight 03

La claridad debe responder “¿cómo voy?”.

El usuario no solo necesita registrar; necesita entender si puede gastar, ahorrar o si debe ajustar antes de cerrar el mes.

02

Exploración inicial

Wireframes para ordenar la experiencia antes del diseño final.

Antes de pulir la interfaz visual, exploré la estructura base de las pantallas principales: inicio, registro de gasto, cuentas o tarjetas y metas de ahorro. La intención era validar jerarquía, flujo y carga de información antes de invertir tiempo en acabado visual.

Inicio / resumen financiero

Exploración de jerarquía para balance, historial, presupuesto y acciones frecuentes.

Registro de gasto

Estructura inicial para capturar monto, fecha, descripción, etiqueta y forma de pago.

Metas / ahorro

Distribución inicial para visualizar avance mensual y objetivos de ahorro.

Tarjetas o cuentas

Exploración para registrar diferentes fuentes de dinero y tarjetas.

03

Evolución visual

Del wireframe a una pantalla más accionable.

La pantalla principal evolucionó desde una estructura base hacia una versión más completa y accionable. Los cambios no buscaban replicar el Figma al 100%, sino mejorar claridad, jerarquía y utilidad conforme avanzó el producto.

Wireframe inicial

Definí estructura, jerarquía y elementos principales del home.

Primera propuesta visual

Exploré una interfaz más limpia, amigable y centrada en la mascota.

Versión final refinada

Ajusté la pantalla para mostrar balance proyectado, salud financiera y metas con mayor claridad.

04

Iteraciones por feedback

Cuatro ajustes que hicieron el producto más realista.

La retroalimentación ayudó a detectar supuestos débiles y convertirlos en decisiones concretas. Cada caso muestra qué existía antes, qué problema apareció, qué cambio apliqué y qué mejoró.

01

Ingresos variables

Antes
El flujo asumía sueldo fijo y fechas de pago predecibles.
Feedback
Una persona por honorarios no podía representar su realidad financiera.
Cambio aplicado
Agregar captura manual de ingresos cuando ocurren.
Resultado
La app sirve para freelancers, ingresos extra y pagos irregulares.
02

Varias cuentas o tarjetas

Antes
El dinero se entendía como un saldo general.
Feedback
El usuario suele separar débito, efectivo, cuentas compartidas o tarjetas.
Cambio aplicado
Preparar el modelo para registrar movimientos por cuenta.
Resultado
La lectura del disponible se vuelve más fiel a la vida diaria.
03

Inicio demasiado extenso

Antes
El home reunía métricas, gráficas, deudas, metas e historial.
Feedback
La pantalla se sentía pesada para una consulta rápida.
Cambio aplicado
Separar lectura profunda y dejar el home para acciones frecuentes.
Resultado
Menos ruido visual y mejor ritmo al abrir la app.
04

Simplificación del inicio

Antes
El primer vistazo intentaba explicar todo el producto.
Feedback
La saturación hacía más difícil decidir qué hacer primero.
Cambio aplicado
Priorizar resumen financiero, historial y botón de registro.
Resultado
La experiencia inicial queda más directa y lista para validar.
05

Solución final

Una app para registrar, entender y anticipar.

La propuesta final reduce el esfuerzo de captura y convierte los movimientos del mes en señales simples para decidir mejor.

06

Resultados, aprendizajes y próximos pasos

Un avance honesto desde una necesidad real.

Hormigon todavía no presume métricas grandes: el aprendizaje principal fue convertir un problema real en un flujo más claro, probarlo con datos cotidianos y preparar el producto para validación externa.

Resultado real Menos Excel se dejó de depender de la hoja de cálculo como única herramienta de seguimiento
Feedback aplicado Más casos ingresos variables y múltiples cuentas dejaron de ser escenarios no considerados
Producto Más claro el inicio se simplificó para reducir saturación y preparar pruebas externas
Próximo paso

Validar con usuarios externos.

Probar el prototipo con personas fuera del contexto familiar para detectar nuevas fricciones.

Próximos pasos

Medir tiempo de registro.

Observar cuánto tarda una persona en registrar gasto, ingreso o deuda sin ayuda.

Evolución

Documentar sistema visual y accesibilidad.

Ordenar componentes, estados, contraste e interacciones antes de una beta cerrada.

Proyecto 02

Innovadora: proyecto web real para empresa industrial

Proyecto secundario de UI/web para una empresa industrial. El enfoque fue ordenar la presencia digital, mantener una identidad visual reconocible y presentar productos, confianza y contacto comercial sin venderlo como un caso UX profundo.

Rol UI/Web Designer
Tipo Proyecto web real
Estilo visual Identidad industrial
Contexto

Una marca industrial necesitaba verse más clara en digital.

La página organiza empresa, productos y contacto para que clientes industriales entiendan rápido qué ofrece Innovadora.

Marca · Productos · Contacto
Objetivo

Convertir el sitio en una vitrina comercial directa.

Prioricé una lectura rápida de servicios, productos y vías de contacto, con una estructura lista para crecer.

Color · Pintura · Sistema visual
Decisiones visuales

Color, contraste y tono industrial como señales de marca.

Home · Productos · Confianza

El diseño queda listo para crecer con fichas técnicas, catálogos, buscador de productos y contenido comercial. El aprendizaje fue equilibrar identidad fuerte con lectura práctica.
